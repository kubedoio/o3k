# O3K v0.2.0 Release Design

**Date:** 2026-06-24  
**Status:** Approved  
**Scope:** First real release — single-machine KVM+SQLite+local-disk, one-liner install

---

## Goal

Ship `v0.2.0` as a production-usable release for single-machine KVM deployments. A user on a bare Ubuntu/Debian or RHEL/Fedora host with `/dev/kvm` should be able to run one command and have a working OpenStack-compatible API within 60 seconds.

```sh
curl -sfL https://get.o3k.io | sh -
```

---

### 2. Install Script (`scripts/install.sh`)

Full rewrite of the existing script. Keeps the same structure but adds KVM-specific preflight, dependency management, config generation, and hardened systemd unit.

#### 2.1 Preflight checks (fail fast, clear errors)

| Check | Failure message |
|-------|----------------|
| Running as root | "Run as root: sudo sh -" |
| Linux OS | "macOS: stub mode only, no service install" |
| `/dev/kvm` exists | "KVM not available — enable virtualisation in BIOS/hypervisor" |
| Required ports free (35357, 8774, 8775, 8776, 8778, 9292, 9696) | "Port N in use by PID X" |
| Sufficient disk (>2GB free in `/var/lib/o3k`) | "Insufficient disk space" |

#### 2.2 Dependency install

Detects package manager (`apt-get` or `dnf`). Installs if not present:
- `libvirt-daemon-system` (or `libvirt-daemon` on RHEL)
- `qemu-kvm`
- `qemu-utils` (for `qemu-img`)
- `virtinst` (optional, for `virt-install` tooling)

Ensures `libvirtd` is enabled and running. Verifies `virsh -c qemu:///system version` succeeds before proceeding.

#### 2.3 Binary install

1. Resolves latest version via GitHub releases API (or uses `O3K_VERSION` env override)
2. Downloads `o3k-linux-<arch>` from release assets
3. Downloads `checksums.txt` and verifies SHA256
4. Installs to `/usr/local/bin/o3k`

#### 2.4 Config generation

Writes `/etc/o3k/config.yaml` (skipped if file already exists, unless `O3K_FORCE_CONFIG=true`):

```yaml
database:
  datastore: "sqlite:///var/lib/o3k/state.db"
keystone:
  jwt_secret: "<random 48-byte base64 from openssl rand>"
  token_ttl: 24h
nova:
  libvirt_mode: real
  libvirt_uri: "qemu:///system"
cinder:
  storage_mode: local
glance:
  storage_mode: local
neutron:
  networking_mode: stub
```

JWT secret generated with `openssl rand -base64 48` at install time. Written to config file (mode 0600, owned by root). Never appears in env vars or unit file.

#### 2.5 Systemd service unit

Written to `/etc/systemd/system/o3k.service`:

```ini
[Unit]
Description=O3K OpenStack Server
After=network-online.target libvirtd.service
Wants=network-online.target
Requires=libvirtd.service

[Service]
Type=simple
ExecStart=/usr/local/bin/o3k --config /etc/o3k/config.yaml
Environment=O3K_DATA_DIR=/var/lib/o3k
Restart=on-failure
RestartSec=5
LimitNOFILE=65535
PrivateTmp=true
NoNewPrivileges=true

[Install]
WantedBy=multi-user.target
```

Service runs as root (required for libvirt socket and KVM device access). See Security section.

#### 2.6 Post-install output

Waits up to 30s for Keystone to respond on `http://localhost:35357/v3`. On success prints:

```
═══════════════════════════════════════════════
  O3K v0.2.0 installed and running
═══════════════════════════════════════════════
  Keystone:  http://localhost:35357/v3
  Nova:      http://localhost:8774/v2.1
  Admin:     admin / <password from /var/lib/o3k/initial-password>

  Quick start:
    export OS_AUTH_URL=http://localhost:35357/v3
    export OS_USERNAME=admin
    export OS_PASSWORD=<password>
    export OS_PROJECT_NAME=default
    export OS_USER_DOMAIN_NAME=Default
    export OS_PROJECT_DOMAIN_NAME=Default
    openstack token issue
═══════════════════════════════════════════════
```

#### 2.7 Env var overrides

| Variable | Default | Purpose |
|----------|---------|---------|
| `O3K_VERSION` | `latest` | Pin to specific release |
| `O3K_INSTALL_DIR` | `/usr/local/bin` | Binary location |
| `O3K_DATA_DIR` | `/var/lib/o3k` | Data/state directory |
| `O3K_SKIP_SERVICE` | `false` | Install binary only, no systemd |
| `O3K_FORCE_CONFIG` | `false` | Overwrite existing config |

### 3. `get.o3k.io` Redirect

**Implementation:** Cloudflare Redirect Rules (no Worker, no code, free tier).

Two rules in Cloudflare dashboard → o3k.io zone → **Rules → Redirect Rules**:

**Rule 1 — latest:**
```
Match:  URL path equals /
Action: Dynamic redirect (302) →
        https://github.com/kubedoio/o3k/releases/latest/download/install.sh
```

**Rule 2 — pinned version:**
```
Match:  URL path wildcard /*
Action: Dynamic redirect (302) →
        https://github.com/kubedoio/o3k/releases/download/${1}/install.sh
```

The `${1}` wildcard capture passes the path segment (e.g. `v0.2.0`) directly into the GitHub releases URL. GitHub's `/releases/latest/download/` always resolves to the most recent published release — no manual update needed when new versions ship.

Routes:
- `curl -sfL https://get.o3k.io | sh -` → latest release install.sh (auto-updates)
- `curl -sfL https://get.o3k.io/v0.2.0 | sh -` → pinned to v0.2.0
- `curl -sfL https://get.o3k.io/v0.3.0 | sh -` → pinned to v0.3.0 (works automatically for future releases)

No deployment artifact needed in the repo. Rules are configured once in the Cloudflare dashboard and never touched again.

### 5. Security Documentation (`docs/security.md`)

Covers:
- **Why root:** libvirt requires root or `libvirt` group; KVM device access; netlink for future networking modes. Same model as k3s, microk8s, minikube.
- **What root access means:** treat the o3k host as you would any bare-metal hypervisor. Don't co-locate with untrusted workloads.
- **Mitigations in v0.2.0:** `NoNewPrivileges`, `PrivateTmp`, JWT secret in config file (mode 0600) not env vars, SHA256-verified binary install, cosign supply chain signatures.
- **Network exposure:** all ports bind `0.0.0.0` by default. Recommended: firewall all OpenStack ports except 35357 from trusted subnets.
- **Roadmap (v0.3):** dedicated `o3k` system user with fine-grained capabilities, `systemd-creds` for secret storage, optional TLS on all service ports.

---

## Out of Scope for v0.2.0

- Multi-node / VXLAN networking (exists in code but not in install script)
- TLS on any port
- PostgreSQL install path (documented as manual in README)
- `o3k agent` mode (install script supports env var but not the focus of this release)
